DEVOPS & WORKFLOWS

Overview

Our delivery pipeline keeps DeadLink reliable and easy to run—locally or in Docker. Here’s how it works.

⚙️

CI checks

Jest test suites run on every push (planned), with smoke tests and analytics/helpers covered. Linting is optional.

Workflow file: .github/workflows/ci-check.yml

🏷️

Releases

Pushing a tag like vX.Y.Z creates a GitHub Release. Release notes are auto-generated unless a body file is present.

Workflow: .github/workflows/release.yml

🐳

Docker images

On release or manual trigger, multi-arch images (amd64/arm64) are built and pushed to GHCR. Tags include the version and latest.

Workflow: .github/workflows/docker.yml → image ghcr.io/thecdrz/deadlink

📰

Website

Docs are served via GitHub Pages. Any change under docs/ triggers a build and deploy.

Workflow: .github/workflows/pages.yml

📸

UI Snapshots

A Puppeteer script renders sample embeds and saves PNGs to screenshots/ and docs/assets/snapshots/ for the site.

Script: scripts/snapshot-embeds.js — Run: npm run snapshots

🚀

Run in Docker

Use GHCR images and set environment variables for Discord and telnet credentials. DEV_MODE=true enables simulated telnet for local demos.

See README → Docker

Release notes

We maintain versioned notes in the repo (e.g., RELEASE_NOTES_v2.7.0.md), and GitHub Releases include summaries.
